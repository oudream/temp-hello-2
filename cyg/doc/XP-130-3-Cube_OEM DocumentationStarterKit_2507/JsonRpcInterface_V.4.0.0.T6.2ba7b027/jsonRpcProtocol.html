5. Introduction

5.1. About

The protocol is based on JSON-RPC 2.0 and running over non-secure WebSocket.

5.2. Notable OSI Layers

  • Layer 2 - Data Link Layer: Ethernet
  • Layer 7 - Application Layer: WebSocket

5.3. JSON-RPC Feature Coverage

The following JSON-RPC standard features are used:

  • regular requests
  • regular response
  • error response
  • synchronous mode (only one request at a time per connection)

The following standard features are currently not supported:

  • batch (multiple requests in the same call are not allowed)
  • asynchronous mode (multiple pending requests in separate calls are not allowed. Strict request-response sequences is required)
  • notification request

5.4. Standard JSON-RPC 2.0

Refer to the Official documentation to get an overview of JSON-RPC 2.0: https://www.jsonrpc.org/specification

In short, JSON-RPC 2.0 defines the following structures:

Request:

{"jsonrpc":"2.0","method":<method>,"params":<params>,"id":<id>}

Response:

{"jsonrpc":"2.0","result":<result>,"id":<id>}

Error:

{"jsonrpc":"2.0","error":{"code":<code>,"message":<message>},"id":<id>}

Standard JSON types are used:

Type Example
String "Hello World!"
Number 42 or 1.23
Boolean true
Null null
Object {"operational/control/request":0}
Array [1,2,3]

5.5. Non-Standard JSON-RPC 2.0 Extensions

5.5.1. File Transfer

In order to handle file transfer in an easy to implement (client point of view) and server resources independent (predefined file segmentation) way, the standard JSON-RPC 2.0 protocol was extended.

7. Read Resource

7.1. JSON-RPC compatiblity

Read Resource complies with the JSON-RPC 2.0 standard.

7.2. Request

{"jsonrpc":"2.0","method":"read","params":{"resource":<resource>},"id":<id>}
Field Type Value
<resource> String resource path
<id> Number unique message id

7.3. Response

{"jsonrpc":"2.0","result":{"resource":<resource>,"value":<value>},"id":<id>}
Field Type Value
<resource> String resource path
<value> [String | Number | Boolean] value of dispatcher resource
<id> Number message id of request

9. Error

9.1. JSON-RPC compatiblity

Error messages comply with the JSON-RPC 2.0 standard.

9.2. Error Response

Generic Error Structure:

{"jsonrpc":"2.0","error":{"code":<code>,"message":<message>}," id":<id>}
Field Type Value
<code> Number error code
<message> String error message

9.3. Error Codes

JSON-RPC 2.0 defines the following error codes:

code message
-32700 Parse error
-32600 Invalid Request
-32601 Method not found
-32602 Invalid params
-32603 Internal error
-32000 to -32099 application specific error codes, not specified here

10. Read File

10.1. JSON-RPC compatiblity

Read File transfer does not comply with the JSON-RPC 2.0 standard. Some parts of the Read File transfer are borrowed from the JSON-RPC 2.0 standard and are thus compatible.

10.2. Request

Request complies with the JSON-RPC 2.0 standard.

{"jsonrpc":"2.0","method":"fileRead","params":{"path":<path>},"id":<id>}
Field Type Value
<path> String file path
<id> Number message id of request

10.3. Response Part 1

Response Part 1 complies with the JSON-RPC 2.0 standard.

{"jsonrpc":"2.0","result":{"path":<path>,"len":<len>},"id":<id>}
Field Type Value
<path> String file path
<len> Number file length
<id> Number message id of request

10.4. Response Part 2

Response Part 2 does not comply with the JSON-RPC 2.0 standard.

After transmitting the first part of the response, the server transmits the file contents in an unknown number of WebSocket frames without any protocol defined for the WebSocket payload. The payload size is limited by the transmit buffer of the server. This means that the client application needs to reassemble the received data. Since the file length was previously transmitted as described in chapter 8.2, the client simply needs to loop the WebSocket receive call until the received number of bytes matches the file length.

The following python code snippet shows how one might implement the file receive functionality:

while rx_counter < file_length:
    data += await websocket.recv()
    rx_counter = len(data)

11. Write File

11.1. JSON-RPC compatiblity

Write File transfer does not comply with the JSON-RPC 2.0 standard. Some parts of the Write File transfer are borrowed from the JSON-RPC 2.0 standard and are thus compatible.

11.2. Request Part 1

Request Part 1 complies with the JSON-RPC 2.0 standard.

{"jsonrpc":"2.0","method":"fileWrite","params":{"path":<path>,"len":<len>},"id":<id>}
Field Type Value
<path> String file path
<len> Number file length
<id> Number message id of request

11.3. Request Part 2

Request Part 2 does not comply with the JSON-RPC 2.0 standard.

After transmitting the first part of the request, the client shall transmit the file contents in an unknown number of WebSocket frames without any protocol defined for the WebSocket payload. There is no limit to the payload size. The client can either split the file into multiple WebSocket frames or send the complete file contained within one frame. After the server has received the number of bytes specified in the first part of the request, the server will send a response.

11.4. Response

Response complies with the JSON-RPC 2.0 standard.

{"jsonrpc":"2.0","result":{"path":<path>},"id":<id>}
Field Type Value
<path> String file path
<id> Number message id of request
